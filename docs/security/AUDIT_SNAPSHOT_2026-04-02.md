# Снимок аудита безопасности (2026-04-02)

Краткий обзор по коду и конфигу; не заменяет penetration test.

## Сильные стороны

- **Edge:** критичные функции опираются на `getAuthenticatedUserId` (реальный Supabase Auth JWT) или service role; anon-only смоки дают **401** на выборке эндпоинтов.
- **CORS:** для браузерных запросов есть allowlist + `RD_ALLOWED_ORIGINS`; для клиентов без `Origin` — `*` (мобильные).
- **Частичный rate limit** в коде функций (`enforceRateLimit*`) на тяжёлых/дорогих путях.
- **Инвайты (чеклист):** криптостойкий токен и урезанный анонимный lookup (см. `HARDENING_SPRINT_CHECKLIST.md`).

## Уязвимости и риски (приоритет)

| Область | Риск | Что сделать |
|--------|------|-------------|
| **Шлюз Supabase** | `verify_jwt = false` у большинства функций — атака доходит до кода | Оставить осознанно только где нужен legacy/хук; везде остальном — жёсткая проверка внутри + CF rate-limit/WAF. |
| **`send-registration-email`** | `hasValidApiKeyOrUser` — валидный **anon** проходит (задумано для триггеров) | Лимиты на CF, секрет/подпись для внутренних вызовов, мониторинг объёма писем. |
| **БД** | Много `SECURITY DEFINER` RPC; встречаются широкие политики (`USING (true)` и т.п. в отдельных миграциях) | Регулярный прогон `scripts/security_audit_rls.sql`, ревью DEFINER на контекст вызывающего и tenant. |
| **Инвайты** | TTL / one-time / хеш токена — по чеклисту не закрыто | Миграции + проверки в RPC. |
| **Инфра** | WAF, алерты по 4xx/5xx, бэкапы/секреты | По `HARDENING_SPRINT_CHECKLIST.md` §4–5. |
| **Клиент** | Anon ключ в `assets/config.json` — ожидаемо публичный; секреты только на сервере | Не класть service role в приложение (проверить сборки). |

## Вывод

**Закрыто не всё из плана харднинга:** Edge-auth сдвинут в правильную сторону, но **база, CF-абьюз, бэкапы и токены приглашений** требуют отдельных итераций. Рекомендуется ручной смок под залогиненным пользователем после каждого изменения функций.
