# Чеклист безопасности для владельца (без разработчика в штате)

Код и миграции живут в репозитории; ниже — что **нельзя** автоматизировать без входа в сервисы под вашим аккаунтом. Делайте по мере сил: раз в квартал лучше, чем никогда.

## 1. GitHub (один раз + периодически)

| Действие | Где |
|----------|-----|
| Включить **Dependabot alerts** (и при желании **Dependabot security updates**) | Репозиторий → **Settings** → **Code security and analysis** |
| **MFA** на аккаунте GitHub | **Settings** (профиль) → **Password and authentication** |
| Просматривать PR от Dependabot (после включения `.github/dependabot.yml`) | Вкладка **Pull requests** — мержить в **`staging`**, не в `main` без проверки на Beta |

Файл `.github/dependabot.yml` создаёт **еженедельные** PR с обновлением зависимостей (npm в корне и `admin`, pub во Flutter). Алерты по уязвимостям настраиваются **отдельно** в Code security (п. выше).

## 2. Supabase (ваш проект)

| Действие | Зачем |
|----------|--------|
| Убедиться, что **все миграции из `supabase/migrations`** применены к единой БД (или выполнен эквивалентный SQL в **SQL Editor**) | Код и база не расходятся |
| **MFA** в настройках аккаунта Supabase | Защита админки проекта |
| Резервное копирование и план проекта — по тарифу Supabase | Восстановление при сбое |

После крупных изменений в `supabase/migrations` — один прогон в SQL Editor или `supabase db push` с вашей машины, если так принято.

## 3. Cloudflare

| Действие | Зачем |
|----------|--------|
| **WAF** и **rate limiting** на чувствительных маршрутах (API, формы) | Периметр вне Supabase |
| Проверить, что **Beta** смотрит на ветку **`staging`**, **Prod** на **`main`** | См. `docs/DEPLOY_RULES.md` / правила в `.cursor/rules` |

## 4. Раз в квартал (15 минут)

1. В **SQL Editor** Supabase выполнить `scripts/security_audit_rls.sql` (полный отчёт и при необходимости блоки «кандидаты» внизу файла).
2. Кратко проверить: нет ли неожиданных **anon + полный доступ** на новых таблицах.

## 5. Когда подключать разработчика

- Ошибки после миграции RLS (403/пустые списки в приложении).
- Нужен пентест или внешний аудит.
- Интеграция централизованных логов и алертов (см. `docs/SECURITY_PROGRAM.md`).

Подробная программа и модель угроз: **`docs/SECURITY_PROGRAM.md`**.
