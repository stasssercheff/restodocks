# 🗄️ ВОССТАНОВЛЕНИЕ И ТЕСТИРОВАНИЕ БАЗЫ ДАННЫХ RESTODOCKS

## 📋 ОБЗОР СКРИПТОВ

| Скрипт | Назначение | Когда использовать |
|--------|------------|-------------------|
| `restore_database.sh` | Восстановление БД из дампа | После создания бэкапа с БД |
| `verify_database.sh` | Проверка структуры БД | После восстановления |
| `test_database.sh` | Функциональное тестирование | После верификации |
| `monitor_database.sh` | Мониторинг состояния | Регулярно или при проблемах |
| `diagnose_database.sh` | Диагностика проблем | При подозрении на проблемы |

## 📚 ПОДРОБНЫЕ ИНСТРУКЦИИ

### 🗄️ `restore_database.sh` - ВОССТАНОВЛЕНИЕ БД

**Что делает:**
- Ищет файл `database.sql.gz` или `database.sql`
- Распаковывает дамп (если нужно)
- Восстанавливает базу данных через `psql`

**Требования:**
- Файл `backup_config.env` с настроенным паролем
- Дамп базы данных (`database.sql.gz`)

**Использование:**
```bash
./restore_database.sh
```

**Что будет запрошено:**
- Подтверждение перезаписи текущей БД

### 🔍 `verify_database.sh` - ПРОВЕРКА СТРУКТУРЫ

**Что проверяет:**
- Подключение к БД
- Наличие основных таблиц
- Индексы и ограничения
- Политики безопасности (RLS)
- Триггеры и функции
- Размеры таблиц

**Использование:**
```bash
./verify_database.sh
```

**Результат:** Подробный отчет о состоянии БД

### 🧪 `test_database.sh` - ФУНКЦИОНАЛЬНОЕ ТЕСТИРОВАНИЕ

**Что тестирует:**
- Основные таблицы и данные
- Связи между таблицами
- Политики безопасности
- Функции и представления
- Целостность данных

**Использование:**
```bash
./test_database.sh
```

**Результат:** Статус прохождения тестов

### 📊 `monitor_database.sh` - МОНИТОРИНГ

**Что показывает:**
- Общую статистику БД
- Активность пользователей
- Состояние заказов и продаж
- Производительность
- Безопасность

**Использование:**
```bash
./monitor_database.sh
```

**Для регулярного мониторинга:**
```bash
# Добавьте в crontab для ежечасного мониторинга
0 * * * * /path/to/monitor_database.sh >> db_monitor.log
```

### 🔧 `diagnose_database.sh` - ДИАГНОСТИКА ПРОБЛЕМ

**Что выявляет:**
- Проблемы подключения
- Отсутствующие таблицы
- Некорректные политики RLS
- Проблемы с индексами
- Нарушения связей

**Использование:**
```bash
./diagnose_database.sh
```

## 🎯 РЕКОМЕНДУЕМЫЙ ПОРЯДОК ДЕЙСТВИЙ

### После создания бэкапа с базой данных:

1. **Подготовка:**
   ```bash
   # Убедитесь что пароль настроен
   ./check_supabase_setup.sh
   ```

2. **Восстановление:**
   ```bash
   ./restore_database.sh
   ```

3. **Проверка:**
   ```bash
   ./verify_database.sh
   ```

4. **Тестирование:**
   ```bash
   ./test_database.sh
   ```

5. **Мониторинг:**
   ```bash
   ./monitor_database.sh
   ```

## 🔧 УСТРАНЕНИЕ ПРОБЛЕМ

### "Пароль не настроен"
```bash
./setup_db_password.sh
```

### "Дамп БД не найден"
```bash
# Распакуйте архив бэкапа
tar -xzf backup_*_COMPLETE.tar.gz
cp backup_*/database.sql.gz ./
gunzip database.sql.gz
```

### "Ошибка подключения"
```bash
# Проверьте настройки
./check_supabase_setup.sh

# Проверьте пароль в Supabase Dashboard
# Возможно нужно сбросить пароль еще раз
```

### "Тесты проваливаются"
```bash
# Запустите диагностику
./diagnose_database.sh

# Возможно нужно переустановить БД
./restore_database.sh
```

## 💾 РЕЗЕРВНОЕ КОПИРОВАНИЕ

### Создание бэкапа БД:
```bash
# Если pg_dump настроен
./backup_all.sh

# Или вручную
pg_dump "$SUPABASE_DB_URL" > manual_backup_$(date +%Y%m%d).sql
```

### Восстановление из резервной копии:
```bash
psql "$SUPABASE_DB_URL" < manual_backup_20241220.sql
```

## 🔐 БЕЗОПАСНОСТЬ

- Пароли хранятся только в `backup_config.env`
- Файл `.env` содержит только API ключи
- Никогда не коммитьте пароли в Git
- Регулярно меняйте пароли БД

## 📞 ПОДДЕРЖКА

Если возникли проблемы:
1. Запустите `./diagnose_database.sh` для диагностики
2. Проверьте логи выполнения скриптов
3. Убедитесь что все файлы находятся в одной директории
4. Проверьте настройки в `backup_config.env`
