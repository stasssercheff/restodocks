# Restodocks — техническое описание (ENISA / Startup Visa)

## 1. Архитектура системы

### 1.1 Клиентский уровень

- Основной клиент: Flutter 3.x (Dart SDK ≥3.5), единая кодовая база для iOS, Android и Web (supabase_flutter, go_router, provider).
- Локальное хранение: shared_preferences, SQLite (sqflite) для крупных JSON-снимков, а также сервисы с ориентацией на офлайн (кэш, гидратация данных заведения).
- Сеть: клиент Supabase с повторными HTTP-запросами; подписки Realtime на `tech_cards` и `products` с отложенным обновлением (~700 мс) и периодическим резервом.

### 1.2 Серверный уровень

- BaaS: Supabase — PostgreSQL, PostgREST, Auth (сессии JWT), Row Level Security (RLS), Realtime, Storage, Edge Functions (Deno/TypeScript).
- Операционные API: Edge Functions для AI-парсинга, закупочных приёмок, биллинга (например проверка Apple IAP), цепочек e-mail.
- Админка: отдельное приложение Next.js (API routes, Supabase на сервере) для операционных задач — не основной UI ресторана.

### 1.3 Гибридная модель данных

- Источник истины: PostgreSQL — реляционные таблицы арендаторов (establishments), сотрудников, продуктов, цен номенклатуры (`establishment_products`), заголовков ТТК (`tech_cards`), нормализованных строк состава (`tt_ingredients`), закупочных документов, переводов, фискальных настроек.
- Структурированные BLOB: JSON/JSONB там, где уместно (секции ТТК, полезные нагрузки документов).
- Клиентское отображение: доменные модели Flutter отображают строки в графы в памяти; себестоимость часто выводится при чтении из текущих цен и структуры рецепта.
- Кэширование: цены/каталог в памяти, обновление по TTL, гидратация по заведению — офлайн-first с сверкой с сервером.

Это гибрид реляционной и документо-подобной схемы: нормализация там, где важна целостность; гибкий JSON там, где удобны блоки UI.

## 2. Динамическая себестоимость (Dynamic Food Cost) — ключевой алгоритм

### 2.1 Источник истины по цене

- Цена за единицу по заведению: `establishment_products` (цена + валюта), обновление через upsert по `(establishment_id, product_id)`.
- История цен: `product_price_history` при изменении эффективной цены сверх малого порога (0.001).

### 2.2 Закупка → обновление цен

- Строки приёмки: фактическое количество, цена за ед., скидка %.
- Эффективная цена за ед. = actualPrice × (1 − discount/100).
- Строки, расходящиеся с номенклатурой, собираются; в зависимости от политики — Edge-воркфлоу или подтверждение на устройстве; выбранные строки вызывают обновление цены номенклатуры (в т.ч. общий каталог на несколько точек).

### 2.3 «Пересчёт» техкарт

- Рецепты ссылаются на продукты и полуфабрикаты (`sourceTechCardId`).
- Отображаемая себестоимость строится гидратацией: `TechCardCostHydrator` для листьев берёт `getEstablishmentPrice` (fallback — `basePrice`), переводит брутто/нетто/шт в эквивалент кг, считает cost = pricePerKg × qty.
- Вложенные ПФ: рекурсия с защитой от циклов (memo + множество resolving); по выходу в граммах и сумме себестоимости вложенного рецепта выводится цена за кг для родительской строки.
- Слои UI (например таблица ТТК) используют согласованное разрешение цен.

Изменение цены по накладной не запускает массовый UPDATE по всем ТТК в SQL; обновляется `establishment_products`. Следующая загрузка, realtime или гидратация пересчитывает затраты — низкая амплификация записи, нет одного устаревающего агрегата в БД.

### 2.4 Производительность и численная устойчивость

- Сложность: O(ингредиенты × глубина вложенности ПФ) с мемоизацией.
- Задержка: доминируют сеть и разбор JSON, не арифметика (double в Dart).
- Точность: числа с плавающей точкой; равенство цен — epsilon 0.001; г↔кг через /1000 — пригодно для операционной себестоимости, не для юридически заверенного аудита.

### 2.5 Краевые случаи

- Нет цены: себестоимость 0 или без изменений до появления цены.
- Штуки (`pcs` / `шт`): масса через `gramsPerPiece` (по умолчанию 50 г).
- Валюта: на строке; свёртка в разных валютах без явного FX не предполагается.

## 3. Технологическая инновация

### 3.1 ИИ в продакшене

- Edge Functions: LLM превращает фото чека/ТТК в структурированный JSON (vision через общий слой провайдера); пакетное извлечение, PDF, списки продуктов, дубликаты, чеклисты.
- Ключи API только на сервере (Edge/Vault); Flutter вызывает только функции.
- On-device OCR (мобильные): Google ML Kit + Apple Vision для низкой задержки без лишнего RTT.

### 3.2 Масштабируемость инженерии

- Провайдеры LLM/OCR меняются за границей функций без переписывания доменного слоя Flutter (в репозитории зафиксированы паттерны интеграции).

### 3.3 Мультиязычность

- UI: единый `localizable.json` по кодам языков; `LocalizationService` — ключи в коде, строки в ассетах; `intl` для форматов.
- Реализованные локали интерфейса: девять (`ru`, `en`, `es`, `kk`, `de`, `fr`, `it`, `tr`, `vi`).
- Контент домена (продукты, ТТК): `TranslationManager` + `TranslationService` — хранение переводов, опционально Google Cloud Translation / MyMemory / ИИ, ручные правки.
- Скрипты поддержки паритета ключей. Нативные сборки включают переводы в бинарник; веб может подхватывать JSON при деплое.

## 4. Безопасность и GDPR (технический уровень)

### 4.1 Аутентификация

- Supabase Auth — JWT; подтверждение e-mail по deep link (implicit flow где применимо).

### 4.2 Авторизация

- RLS на таблицах арендаторов; политики для authenticated и области заведения.
- RPC `check_establishment_access(establishment_id)` — подписка, промо, права; в миграциях жёстко настроены GRANT/REVOKE.
- Чувствительные сценарии (закупки, документы заказов) — через Edge Functions с серверной валидацией и при необходимости service role.

### 4.3 GDPR (без юридических консультаций)

- Многоязычная политика конфиденциальности в приложении (`legal_texts.dart`).
- Мультитенантность с разделением данных по заведению.
- Хранение: в тексте политики — ссылки на бухгалтерские/законодательные сроки; резервное копирование и экспорт — организационный уровень.

## 5. Масштабируемость и новые рынки

### 5.1 Валюта

- Валюта по умолчанию для заведения и валюта строк номенклатуры; централизованные списки ISO — расширение рынка в основном данные + UX, без ломки схемы.

### 5.2 Налоги (НДС / IVA / регионы)

- Фискальные пресеты: версионируемый JSON (`world_tax_presets.json`) — регионы, списки ставок НДС, режим цены (`tax_included` / `tax_excluded`), доп. налоги.
- Переопределения по заведению: регион, override НДС, режим цены; `effectiveVatPercent` = пресет ± override.
- Новая юрисдикция (например Испания / IVA): добавление блока региона, подписей, переводов — инкрементальная доработка существующего модуля.

### 5.3 Нагрузка

- PostgreSQL + Edge — стандартное масштабирование (пул соединений, реплики при необходимости).
- Debounce и TTL на клиенте снижают лавину чтений при многих устройствах.

## 6. Готовность к эксплуатации

- Версионируемые SQL-миграции; инструменты i18n; логирование в Edge (`log-system-error`); проверка оплаты Apple.
- Драйвер ККТ в коде не подключён (`isKktDriverConfigured == false`) — очередь фискальных операций есть; полная интеграция с ККТ — отдельный этап.
